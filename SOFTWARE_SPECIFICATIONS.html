Software Specifications

5.1 Firmware

Language: C++ / Arduino

Device: ESP32

Libraries Used

  • WiFi.h: Network Connectivity
  • PubSubClient.h: MQTT Protocol
  • ArduinoJson.h: Data Serialization

Firmware Logic Flow

1. Initialize Serial & GPIO
2. Connect WiFi & MQTT Broker
3. Loop:
  → Read Sensors (Moisture, Rain)
  → Map Analog to %
  → Publish JSON to Topic
  → Listen for Pump Commands

5.2 Backend

Cloud Platform

  • Compute: Railway (FastAPI), Vercel (Next.js)
  • Database: Supabase (PostgreSQL)

API Architecture

  • AI Service (FastAPI):
    • /api/ai/interpret (Analysis)
    • /api/ai/chat (Gemini LLM)
    • /api/ai/predict (Forecasting)
  • Frontend (Next.js): Supabase Client for CRUD

Database Schema

  • Core: User, Device, Zone
  • Data: SensorData, PumpLog, Alert
  • Ops: IrrigationSchedule, Prediction

5.3 Frontend / Dashboard

Framework: Next.js 16

Style: Tailwind CSS 4

UI Flow & Features

  • Auth: Secure Login via Supabase
  • Dashboard: Real-time sensor cards (Moisture, NPK, pH)
  • AI Insights: Integrated Chatbot Assistant
  • Control: Manual Pump/Fertilizer overrides
  • Reports: Data Visualization (Recharts) & PDF Export
6

IoT Data Pipeline

1. Data Acquisition

  • Devices: ESP32 + Sensors
  • Interval: Every 5 seconds
  • Protocol: MQTT/TCP

2. Pre-processing

  • Edge: Map Analog (0-4095) → Standard Units
  • Cloud: Pydantic Validation & Cleaning

3. Storage & Logs

  • DB: Supabase PostgreSQL (Time-series)
  • Logs: Action audits in PumpLog table
  • History: Persistent sensor_data records

4. Security

  • Transport: SSL/TLS Encryption
  • Access: Supabase RLS (Row Level Security)
  • Secrets: Environment Variables (.env)
7

Machine Learning & AI Model

Core Model Architecture

Model: Google Gemini 1.5 Flash

7.1 Problem Type

  • NLP & Reasoning: Contextual advisory via LLM interpretation.
  • Time-Series Regression: Predictive forecasting of moisture/nutrient trends.
  • Classification: Real-time soil health & stress level categorization.
  • Optimization: Prescriptive resource allocation across multi-zones.

7.3 Architecture Details

  • Type: Multimodal Transformer (LLM)
  • Role: Contextual reasoning & Decision Support
  • Auxiliary: Custom Python Logic for Zone Optimization

Data & Methodology

7.2 Dataset / Context

  • Input Source: Real-time telemetry (Live Sensor Data)
  • Knowledge Base: LLM General World Knowledge + Farm Expenses
  • Context Window: Dynamic injection of current farm state

7.4 Training Methodology

  • Technique: In-Context Learning (Prompt Engineering)
  • Prompt Strategy: System prompts injected with live JSON data
  • Params: Temp 0.7, MaxTokens 800, TopP 0.9

Performance & Deployment

7.5 Metrics

  • Latency: < 2 seconds (optimized via Flash model)
  • Reliability: 100% (Rule-Based Fallback System)
  • Accuracy: Context-Aware qualitative relevance

7.6 Deployment Strategy

  • Platform: Cloud Microservice (Docker/FastAPI)
  • Update: CI/CD Pipeline (Git → Railway)
  • Optimization: Stateless Request/Response Architecture
