Software Specifications

5.1 Firmware

Language: C++ / Arduino

Device: ESP32

Libraries Used

  • WiFi.h: Network Connectivity
  • PubSubClient.h: MQTT Protocol
  • ArduinoJson.h: Data Serialization

Firmware Logic Flow

1. Initialize Serial & GPIO
2. Connect WiFi & MQTT Broker
3. Loop:
  → Read Sensors (Moisture, Rain)
  → Map Analog to %
  → Publish JSON to Topic
  → Listen for Pump Commands

5.2 Backend

Cloud Platform

  • Compute: Railway (FastAPI), Vercel (Next.js)
  • Database: Supabase (PostgreSQL)

API Architecture

  • AI Service (FastAPI):
    • /api/ai/interpret (Analysis)
    • /api/ai/chat (Gemini LLM)
    • /api/ai/predict (Forecasting)
  • Frontend (Next.js): Supabase Client for CRUD

Database Schema

  • Core: User, Device, Zone
  • Data: SensorData, PumpLog, Alert
  • Ops: IrrigationSchedule, Prediction

5.3 Frontend / Dashboard

Framework: Next.js 16

Style: Tailwind CSS 4

UI Flow & Features

  • Auth: Secure Login via Supabase
  • Dashboard: Real-time sensor cards (Moisture, NPK, pH)
  • AI Insights: Integrated Chatbot Assistant
  • Control: Manual Pump/Fertilizer overrides
  • Reports: Data Visualization (Recharts) & PDF Export
6

IoT Data Pipeline

1. Data Acquisition

  • Devices: ESP32 + 7-in-1 Soil Sensor (RS485/Modbus).
  • Sensors: NPK, pH, EC, Moisture, Rain, Battery.
  • Interval: Telemetry pulse every 5 seconds.
  • Protocol: MQTT over TCP (Aedes Broker).

2. Pre-processing

  • Edge: Analog-to-Digital mapping & Modbus register conversion.
  • Cloud: JSON parsing, validation, & "Smart Fallbacks" for missing data.

3. Storage & Logs

  • DB: Supabase PostgreSQL (Time-series optimization).
  • Audits: Detailed PumpLog & Alert history.
  • History: Persistent records for forecasting.

4. Security Measures

  • Transport: Encrypted Proxy for MQTT; HTTPS for APIs.
  • Access: Supabase RLS (Row Level Security).
  • Monitoring: Real-time socket.io heartbeat.
7

Machine Learning & AI Model

Core Model Architecture

Model: Google Gemini 1.5 Flash

7.1 Problem Type

  • Classification: Real-time soil health & stress categorization.
  • Regression: Moisture & nutrient trend forecasting.
  • NLP Reasoning: Contextual advisory via LLM interpretation.
  • Optimization: Weather-aware resource allocation.

7.3 Architecture Details

  • Type: Multimodal Transformer (LLM).
  • Framework: FastAPI (Backend), Gemini API.
  • Hybrid Logic: LLM Reasoning + Deterministic Rule Fallbacks.

Data & Methodology

7.2 Dataset

  • Source: Real-time telemetry (5s pulses) & historical trends via Supabase.
  • Size: ~17,280 readings/day per device (5s interval); scaling to 1M+ records.
  • Data Labeling Process: Automated contextual tagging (Soil Health, Stress) by AI Backend.
  • Dataset Versioning: SQL schema-based versioning & daily database snapshots.

៧.៤ ព័ត៌មានលម្អិតនៃការបង្វឹក (Training Details)

AI ប្រើប្រាស់វិធីសាស្ត្រ In-Context Learning ដោយបញ្ចូលទិន្នន័យ JSON Telemetry ទៅក្នុង Prompt សម្រាប់ការវិភាគ និងផ្ដល់អនុសាសន៍ក្នុងពេលជាក់ស្តែង (Real-Time Inference)។

  • វិធីសាស្ត្រ (Methodology): In-Context Learning (Prompt Engineering)។ ទិន្នន័យ JSON telemetry ត្រូវបានបញ្ចូលទៅក្នុង context window របស់ LLM សម្រាប់ការសម្រេចចិត្ត។
  • ប៉ារ៉ាម៉ែត្រ (Hyperparameters): Temperature: 0.7, TopP: 0.9, MaxOutputTokens: 800។
  • ហេដ្ឋារចនាសម្ព័ន្ធ (Hardware): ប្រើប្រាស់ Distributed Cloud Inference (Google Cloud TPU/GPU) តាមរយៈ Gemini API និង Local GPU Environment សម្រាប់ការធ្វើតេស្ត។

7.4 Training Details

  • Methodology: In-Context Learning (Prompt Engineering). Live JSON telemetry is injected into the LLM context window for real-time inference.
  • Hyperparameters: Temperature: 0.7 (optimal for reasoning balance), TopP: 0.9, MaxOutputTokens: 800.
  • Hardware Used: Distributed Cloud Inference (Google Cloud TPU/GPU infrastructure) via Gemini API; Local Dev on local GPU environments.
  • Inference Strategy: System-level prompt steering with zero-shot and few-shot agricultural safety constraints.

Performance & Deployment

7.5 Model Performance Metrics

  • Latency: < 2 seconds per inference.
  • Reliability: 100% via smart fallback to rule-based logic.
  • Inference: High-speed optimized Flash model.

7.6 Model Deployment

  • Cloud Hosting: The AI "brain" lives on a cloud server (Railway), making it accessible to your farm from anywhere.
  • Automatic Updates: The system updates instantly whenever code is saved—no manual uploading needed.
  • Remote Management: You can update the AI's logic in the cloud without ever having to plug in or reprogram the physical ESP32 device.

9.2 Integration Tests

Integration testing validated interactions between system layers and ensured correct end-to-end system behavior. These tests confirmed seamless data flow from physical sensors through the edge device and cloud infrastructure to the user interface and back to physical actuation.

Key integration scenarios included:

  • End-to-end data transmission: sensors → ESP32 → MQTT → cloud backend → database → dashboard
  • Sensor Processing: Successful processing of data originating from the LoRaWAN 7-in-1 soil sensor into cloud analytics and AI decision logic
  • Control Loop: Transmission of AI-generated irrigation and fertilization commands back to the ESP32 for execution via relay modules
  • Weather Optimization: Proper integration of weather data from OpenWeatherMap into AI-based optimization logic
  • Financial Sync: Automatic updating of expense tracking records following pump activation events
Test Result / លទ្ធផលតេស្ត៖ These integration tests confirmed that the system operates as a closed-loop IoT–AI architecture without data loss, synchronization errors, or command execution failures.
