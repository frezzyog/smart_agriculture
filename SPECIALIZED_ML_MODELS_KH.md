# 🧠 ម៉ូដែល ML ឯកទេស (Specialized ML Models)
**ផ្នែក: គណនា និងព្យាករណ៍ (Compute & Predict)**

នៅក្នុងប្រព័ន្ធរបស់យើង ក្រៅពីការប្រើប្រាស់ Google Gemini យើងមានម៉ូដែល ML ចំនួន ៣ ផ្សេងទៀតដែលមិនមែនជា Generative AI តែវាផ្តោតលើការវិភាគទិន្នន័យសុទ្ធសាធ (Mathematics & Statistics) ដើម្បីធានាបាននូវភាពត្រឹមត្រូវតាមបែបកសិកម្ម៖

---

### ក. Irrigation Predictor (ការព្យាករណ៍ការស្រោចស្រព)
ម៉ូដែលនេះប្រើប្រាស់ការវិភាគ **Time-series Analysis** ដើម្បីព្យាករណ៍ពីតម្រូវការទឹកនាពេលអនាគត។
*   **បច្ចេកទេស**: វាមិនប្រើការស្មានទេ ប៉ុន្តែវាគណនា **អត្រារំហួត (Moisture Depletion Rate)**។
*   **តួនាទី**: បើទិន្នន័យបច្ចុប្បន្នបង្ហាញថាសំណើមដីធ្លាក់ចុះ ១០% ក្នុងមួយថ្ងៃ ម៉ូដែលនេះនឹងប្រាប់ថា៖ *"នៅ ៣ ថ្ងៃទៀត សំណើមដីនឹងធ្លាក់ដល់ ៤០% (កម្រិតគ្រោះថ្នាក់) ដូច្នេះប្រព័ន្ធត្រូវត្រៀមបូមទឹក"*។
*   **បច្ចេកវិទ្យាបន្ថែម**: យើងក៏ប្រើ **Facebook Prophet** សម្រាប់ការព្យាករណ៍ទិន្នន័យដែលមានលក្ខណៈរដូវកាល (Seasonality) ផងដែរ។

### ខ. Fertilizer Predictor (ការព្យាករណ៍ជាតិជី)
ម៉ូដែលនេះតាមដាន និងព្យាករណ៍តុល្យភាពសារធាតុចិញ្ចឹមនៅក្នុងដី (NPK)។
*   **បច្ចេកទេស**: វាគណនាអត្រាដែលដំណាំស្រូបយកសារធាតុ **Nitrogen (N), Phosphorus (P), និង Potassium (K)** តាមពេលវេលា។
*   **តួនាទី**: វាជួយការពារកុំឱ្យរុក្ខជាតិខ្វះជី ដោយវាអាចដឹងមុនថា *"ជាតិអាសូត (N) នឹងអស់នៅសប្តាហ៍ក្រោយ"* ដោយផ្អែកលើល្បឿននៃការប្រើប្រាស់បច្ចុប្បន្ន។

### គ. Zone Optimizer (ការបង្កើនប្រសិទ្ធភាពតំបន់)
ម៉ូដែលនេះគឺជាអ្នកសម្រេចចិត្ត (Decision Maker) នៅពេលដែលធនធាន (ទឹក/ជី) មានកម្រិត។
*   **បច្ចេកទេស**: វាប្រើរូបមន្ត **Priority Scoring (Urgency × Crop Priority)**។
*   **តួនាទី**: ប្រសិនបើយើងមានទឹកក្នុងស្តុកតិច ម៉ូដែលនេះនឹងសម្រេចចិត្តថា *"តំបន់ណាមានអាទិភាពជាង?"*។
    *   ឧទាហរណ៍៖ តំបន់ A (ស្ងួតខ្លាំង + ដំណាំមានតម្លៃខ្ពស់) នឹងទទួលបានទឹកមុនតំបន់ B (សំណើមល្មម)។

### ⚠️ បញ្ហាប្រឈមក្នុង ML និងដំណោះស្រាយ (Challenges & Solutions)
គណៈកម្មការអាចនឹងសួរថា "តើមានបញ្ហាអ្វីខ្លះពេលប្រើ ML ក្នុងកសិដ្ឋាន?" នេះគឺជាចំណុចសំខាន់ៗ៖

១. **Cold Start Problem (បញ្ហាកង្វះទិន្នន័យដំបូង)**:
*   **បញ្ហា**: ពេលទើបដំឡើងថ្មី ប្រព័ន្ធមិនទាន់មានទិន្នន័យចាស់ៗសម្រាប់ព្យាករណ៍ (Time-series) ទេ។
*   **ដំណោះស្រាយ**: យើងប្រើ **Deterministic Fallback Model** (ច្បាប់គណិតវិទ្យាថេរ) ដើម្បីដំណើរការជាបណ្តោះអាសន្ន រហូតដល់ប្រព័ន្ធប្រមូលទិន្នន័យបានគ្រប់គ្រាន់សម្រាប់ឱ្យ ML រៀន។

២. **Data Quality (គុណភាពទិន្នន័យ)**:
*   **បញ្ហា**: បើសិនស័រខូច ឬមានការរំខាន (Noise) វានឹងធ្វើឱ្យ ML ព្យាករណ៍ខុស។
*   **ដំណោះស្រាយ**: យើងប្រើ **Data Validation Layer** ដើម្បីឆែកមើលទិន្នន័យខុសប្រក្រតី (Outliers) មុននឹងបញ្ជូនទៅ ML និងប្រើ **Median Filter** ដើម្បីកាត់បន្ថយការរំខាន។

៣. **Environment Dependencies (ការពឹងផ្អែកលើបច្ចេកវិទ្យា)**:
*   **បញ្ហា**: បណ្ណាល័យ ML ធំៗ (ដូចជា Prophet) មិនអាចដើរលើ ESP32 បានទេ។
*   **ដំណោះស្រាយ**: យើងរៀបចំវាជា **Hybrid Model** ដោយយកការគណនាធំៗដាក់លើ Cloud (Server) ហើយរក្សាទុកការគណនាស្រាលៗ និងច្បាប់សុវត្ថិភាពនៅជាប់នឹងឧបករណ៍ (Edge)។

### �️ ការគ្រប់គ្រងហានិភ័យនៃ ML (Risk Management)
គណៈកម្មការអាចនឹងសួរថា "តើមានហានិភ័យអ្វីខ្លះ បើ ML ធ្វើការខុស?" នេះគឺជាចំណុចគ្រោះថ្នាក់ និងវិធីការពារ៖

១. **Dependency Risk (ហានិភ័យការដាច់អ៊ីនធឺណិត)**:
*   **ហានិភ័យ**: បើ Cloud AI មិនអាចទាក់ទងបាន តើដំណាំនឹងងាប់ឬ?
*   **ការការពារ**: យើងប្រើ **Local Edge Intelligence**។ ESP32 មាន 'ច្បាប់សុវត្ថិភាពខ្ពស់បំផុត' ដែលនឹងដំណើរការដោយស្វ័យប្រវត្តិតាមស្ថានភាពជាក់ស្តែង ទោះគ្មានអ៊ីនធឺណិតក៏ដោយ។

២. **Over-Automation Risk (ហានិភ័យនៃការបញ្ជាខុស)**:
*   **ហានិភ័យ**: ML អាចនឹងបញ្ជាឱ្យស្រោចទឹកខ្លាំងពេក រហូតដល់រលួយឫស ឬបូមជីច្រើនពេកដល់ដីខូច។
*   **ការការពារ**: យើងមាន **Fail-Safe Thresholds**។ យើងកំនត់កំរិតអតិបរមា (Max Limits) នៅក្នុងកូដ Hardware។ ទោះ AI បញ្ជាឱ្យបើកទឹក ១ ម៉ោង ក៏ Hardware នឹងកាត់ផ្តាច់ត្រឹម ២០ នាទី បើវាលើសពីតម្រូវការបច្ចេកទេស។

៣. **Sensor Integrity Risk (ហានិភ័យសិនស័រផ្តល់តម្លៃមិនពិត)**:
*   **ហានិភ័យ**: សិនស័រវាស់សំណើមអាចនឹងគាំងនៅតម្លៃ ០% ធ្វើឱ្យ AI គិតថាដីស្ងួត ហើយបូមទឹកមិនឈប់។
*   **ការការពារ**: យើងប្រើ **Cross-Verification**។ ប្រព័ន្ធនឹងឆែកមើលសិនស័រផ្សេងទៀត (ដូចជា សំណើមបរិយាកាស និងសីតុណ្ហភាព)។ បើទិន្នន័យមិនស៊ីគ្នា (ឧទាហរណ៍៖ សំណើមដី ០% តែសំណើមខ្យល់ ៩០%) ប្រព័ន្ធនឹងផ្ញើសារអាសន្នទៅ **Telegram** ឱ្យម្ចាស់កសិដ្ឋានឆែកមើលភ្លាម។

### � ហានិភ័យដែលមិនទាន់បានការពារ (Remaining Risks & Future Improvements)
ជាការឆ្លើយតបយ៉ាងស្មោះត្រង់ទៅកាន់គណៈកម្មការ នេះគឺជាហានិភ័យបច្ចេកទេសមួយចំនួនដែលយើងកំពុងសិក្សាដើម្បីកែលម្អបន្ថែម៖

១. **Dry Pump Run (ហានិភ័យបូមទាំងគ្មានទឹក)**:
*   **ស្ថានភាព**: បច្ចុប្បន្នយើងមិនទាន់មាន **Level Sensor** ក្នុងអាងទឹកទេ។ បើទឹកក្នុងអាងអស់ ម៉ូទ័រនឹងបន្តដើរដែលនាំឱ្យឆេះម៉ូទ័រ។
*   **ដំណោះស្រាយនាពេលអនាគត**: បន្ថែមសិនស័រវាស់កម្រិតទឹកក្នុងអាង និង **Flow Sensor** ដើម្បីបញ្ញាក់ថាមានទឹកហូរចេញមកពិតមែន។

២. **Actuator Failure Feedback (កង្វះមតិត្រឡប់ពីម៉ូទ័រ)**:
*   **ស្ថានភាព**: បើ Relay គាំងនៅតម្លៃ ON (ជាប់ជានិច្ច) ប្រព័ន្ធនឹងបន្តបូមទឹកដោយមិនឈប់ ទោះ AI បញ្ជាឱ្យបិទក៏ដោយ។
*   **ដំណោះស្រាយនាពេលអនាគត**: ប្រើ **Current Sensor** ដើម្បីវាស់ចរន្តអគ្គិសនីរបស់ម៉ូទ័រ។ បើ AI បិទហើយ តែនៅមានចរន្តដើរ នោះប្រព័ន្ធនឹងកាត់ផ្តាច់តាមរយៈ Switch សុវត្ថិភាពផ្សេងទៀត។

៣. **MQTT Security (សុវត្ថិភាពបណ្តាញ)**:
*   **ស្ថានភាព**: បច្ចុប្បន្នការបញ្ជាតាម MQTT មិនទាន់មានការប្រើប្រាស់ Password ឬការបិទបាំងទិន្នន័យ (Encryption) ខ្លាំងក្លានៅឡើយទេ។
*   **ដំណោះស្រាយនាពេលអនាគត**: អនុវត្ត **SSL/TLS Encryption** និងការកំណត់អត្តសញ្ញាណ (Authentication) ឱ្យបានហ្មត់ចត់បំផុត ដើម្បីបង្ការជនខិលខូចលួចបញ្ជាកសិដ្ឋាន។
